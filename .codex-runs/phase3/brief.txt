<context>
Repo working root = launch dir (git repo, branch main). Review the CURRENT source of
four Supabase Edge Functions as hardened by commit a916ac7a ("harden(edge): fail closed
webhook and MFA reset paths"). The security goal of each:

- verify-mfa-reset : must NOT verify any code. The endpoint should be effectively
  DISABLED (e.g. return a generic 410), because it contained an auth bypass that
  accepted any 6-digit TOTP via a format-only fallback. No unconditional true, no
  format-only 6-digit fallback, no user enumeration, no PII logs.

- stripe-webhook   : must verify the Stripe signature on the RAW body via
  stripe.webhooks.constructEvent. STRIPE_WEBHOOK_SECRET and the stripe-signature
  header are BOTH required. No unsigned "development" fallback. Invalid/missing
  signature must fail closed (non-200).

- tiktok-webhook   : POST must verify a signature/secret (Authorization header
  compared to TIKTOK_WEBHOOK_SECRET, ideally constant-time). Missing secret or
  signature, or mismatch, must fail closed (401). No full body logging.

- whatsapp-webhook : POST must verify x-hub-signature-256 via HMAC-SHA256 of the raw
  body using the app secret (WHATSAPP_APP_SECRET / META_APP_SECRET / FACEBOOK_APP_SECRET).
  Missing secret/signature or HMAC mismatch must fail closed (401). The GET verify-token
  handshake must NOT relax POST security. Catch blocks must NOT return fake success
  (200) when processing failed. No PII logs (phone numbers, contact names, message text).

"Fail closed" = returns an error status (400/401/500) and does NOT perform the
protected action.
</context>

<task>
PHASE 3 — Verify the SECURITY BEHAVIOR of each of the four functions. Read each file in
FULL (supabase/functions/<name>/index.ts). For each requirement below, cite the exact
file:line and code that satisfies it, OR state NOT FOUND and explain the risk.

### A. verify-mfa-reset/index.ts
1. No unconditional `return success/true/verified` path that grants verification.
2. No format-only 6-digit fallback that accepts any 6-digit code as valid.
3. Invalid/missing challenge fails closed (no verification granted).
4. No user enumeration (distinct responses for "user exists" vs "not").
5. No email / factor id / code PII in logs or response body.
6. Confirm the endpoint is effectively disabled (e.g. POST -> generic 410/4xx with a
   constant body) and does NOT perform any password reset or factor verification.

### B. stripe-webhook/index.ts
1. No unsigned development fallback (e.g. "if no secret, trust the body").
2. STRIPE_WEBHOOK_SECRET required; missing -> fail closed (500/4xx).
3. stripe-signature header required; missing -> fail closed (400/4xx).
4. Event obtained ONLY via stripe.webhooks.constructEvent(rawBody, sig, secret).
5. Invalid signature -> fail closed (400). No catch that turns signature errors into 200.
6. Raw body used for verification (not a re-stringified parsed body that would break
   the signature).

### C. tiktok-webhook/index.ts
1. POST signature verification required (Authorization header vs TIKTOK_WEBHOOK_SECRET).
2. Missing secret -> fail closed (401/500) BEFORE processing.
3. Missing signature -> fail closed (401).
4. Invalid/mismatched signature -> fail closed (401). Ideally constant-time compare.
5. No full payload/body JSON logging.
6. GET handshake (if any) does not bypass POST security.

### D. whatsapp-webhook/index.ts
1. POST requires x-hub-signature-256 header.
2. HMAC-SHA256 verification of the RAW body using the app secret. Identify which env
   var(s) are accepted (WHATSAPP_APP_SECRET / META_APP_SECRET / FACEBOOK_APP_SECRET).
3. Missing secret -> fail closed (401/500).
4. Missing signature -> fail closed (401).
5. HMAC mismatch -> fail closed (401).
6. Catch blocks do NOT return fake 200 success when message processing failed.
7. GET verify-token handshake does NOT bypass POST signature security (the GET path
   may return 200 ONLY for the correct hub.verify_token challenge; POST is still
   enforced separately).
8. No PII logs: scan for console.log/error of phone numbers (wa.me, from/to fields),
   contact names, message text, recipient ids.

### E. Secrets
- Confirm no secret VALUES are printed by any of the four files (only env var reads).
